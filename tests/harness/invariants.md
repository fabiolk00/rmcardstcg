# Invariantes do harness de admin — RM Cards

Catalogo das 13 invariantes que TODA operacao manual de admin deve preservar. As
features do ledger (`admin_features.json`) referenciam estes ids no campo
`invariants[]`. As regras sao derivadas do schema (`prisma/schema.prisma`), dos
CHECKs (`prisma/test-schema-supplement.sql`) e das funcoes de dominio
(`lib/data/*`, `lib/cart/totals.ts`).

| id                             | regra                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `cents-only`                   | Todo valor monetario e Int de centavos (colunas `*Cents` / `*_cents`). Nunca float, nunca string formatada no banco. Aritmetica de dinheiro sempre inteira.                                                                                                                                                                                                                                                                                                                          |
| `final-price-derived`          | O preco final do produto e DERIVADO por `finalPriceCents(p) = round(priceCents * (1 - discountPct/100))` (`lib/data/pricing.ts`). NUNCA existe coluna persistida de preco final; so `price_cents` e `discount_pct` ficam no banco.                                                                                                                                                                                                                                                   |
| `reserved-le-stock`            | `0 <= products.reserved <= products.stock` sempre (CHECKs `products_reserved_nonneg_chk`, `products_stock_nonneg_chk`, `products_reserved_le_stock_chk`). Disponivel para venda = `stock - reserved`. Nenhuma mutacao pode violar isso.                                                                                                                                                                                                                                              |
| `reserve-lifecycle-idempotent` | O ciclo reservar -> baixar -> estornar/repor e idempotente via flags do pedido (`Order.stockReserved` / `Order.stockCommitted`): reserva no checkout (`reserveStock`, `stockReserved=true`); baixa na confirmacao (`commitStock`, `stockCommitted=true`); estorno no cancelamento (`releaseStock`, `stockReserved=false`); reposicao no refund de pago (`restockUnits`, `stockCommitted=false`). Repetir um efeito ja aplicado = no-op (guard pela flag, via CAS no proprio pedido). |
| `audit-same-tx`                | TODA mutacao de admin grava EXATAMENTE uma linha em `audit_log` na MESMA transacao da mutacao (`writeAuditLog(tx, ...)`). Rollback da mutacao = rollback da auditoria. `action`/`entityType` sao os enums reais; `before`/`after` sao snapshots jsonb do dominio.                                                                                                                                                                                                                    |
| `webhook-idempotent`           | Eventos de provedor sao idempotentes: `webhook_events (provider, eventId)` UNIQUE + `processedAt` (NULL = recebido, nao concluido). Reprocessar o mesmo evento nao reaplica efeito. (Dominio de billing; aqui so como ancora de invariante.)                                                                                                                                                                                                                                         |
| `order-state-machine`          | Transicoes de status seguem as maquinas de estado. Pagamento (`PaymentStatus`): `pending->paid`, `pending->cancelled`, `paid->cancelled`; `paid`/`cancelled` terminais (nunca voltam a `pending`; `cancelled` nunca vira `paid`). Envio (`ShippingStatus`, `SHIPPING_TRANSITIONS`): `pending->sent`, `pending->cancelled`, `sent->delivered`, `sent->cancelled`; `delivered`/`cancelled` terminais. Pulos/regressoes sao barrados (`invalid_transition`), sem efeito colateral.      |
| `coupon-coherence`             | Coerencia tipo<->campo do cupom (CHECK `coupons_type_value_chk`): `type='percent'` exige `percent_off` em 1..100 e `value_cents` NULL; `type='fixed'` exige `value_cents > 0` e `percent_off` NULL. `toCouponData` zera o campo do outro tipo.                                                                                                                                                                                                                                       |
| `coupon-code-ci-unique`        | Codigo de cupom e unico CASE-INSENSITIVE via indice funcional `UNIQUE (LOWER(code))` (`coupons_code_key`). Criar/editar para um codigo que ja existe em qualquer caixa => P2002 (erro tratado), nunca duas linhas. Codigo normalizado em UPPER no dominio (`normalizeCouponCode`).                                                                                                                                                                                                   |
| `coupon-redeem-limits`         | Limites de redencao respeitados atomicamente: global `maxRedemptions` (increment CAS `WHERE redeemed_count < max`), `perUserLimit` (recontagem sob advisory lock por (cupom,usuario)), e idempotencia por pedido (`coupon_redemptions.order_id` UNIQUE). `redeemedCount >= 0` e `<= maxRedemptions`.                                                                                                                                                                                 |
| `coupon-delete-guard`          | Cupom com QUALQUER redencao NUNCA pode ser excluido (FK `coupon_redemptions.coupon_id` onDelete: Restrict). `deleteCoupon` conta redencoes e recusa (`in_use`); corrida vira P2003 tratado. Cupom usado deve ser INATIVADO (`setCouponActive(false)`), nao excluido.                                                                                                                                                                                                                 |
| `totals-formula`               | `Order.totalCents = subtotalCents - discountCents - couponDiscountCents + shippingCents`. `subtotalCents` = soma de `priceCents*qty`; `discountCents` = desconto de produto; `couponDiscountCents >= 0` (CHECK). Mercadoria = `subtotal - discount`. Frete (`lib/cart/totals.ts`): GRATIS se mercadoria >= R$299 (`FREE_SHIPPING_THRESHOLD_CENTS=29900`) ou == 0; senao flat `FLAT_SHIPPING_CENTS=2500`.                                                                             |
| `pure-client-safe`             | As funcoes de dominio client-safe (`lib/data/pricing.ts`, `lib/data/orderTransitions.ts`, `lib/cart/totals.ts`) sao PURAS e NAO importam `prisma` nem nada server-only, para nao arrastar o driver do banco ao bundle do cliente. As maquinas de transicao (PAYMENT/SHIPPING) sao a unica fonte de verdade das transicoes legais.                                                                                                                                                    |
