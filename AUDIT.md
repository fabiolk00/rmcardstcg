# RM Cards — Auditoria multiagente: relatório e estado

Auditoria de correção/concorrência conduzida com o setup de `ORCHESTRATION.md`
(4 auditores de domínio em paralelo — billing, stock, users, N+1 — + QA
adversarial). Cada fix passa pelo `scripts/qa-gate.sh` (typecheck · lint ·
format · `test:nplusone` · build) e os de domínio têm prova contra um Postgres
real (opt-in por `TEST_DATABASE_URL`; ver `tests/nplusone/README.md`).

## 🔓 Abertos / adiados (com motivo)

| Achado                                                            | Sev. | Por que NÃO foi aplicado agora                                                                                                                                                                                                                       | Abordagem proposta                                                                                                                                                   |
| ----------------------------------------------------------------- | ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `per_user_limit` sem `UNIQUE(coupon_id, user_id)` de respaldo     | 🟠   | **Sem bug vivo**: o fluxo usa `Serializable` + retry; o SSI do Postgres vale mesmo sob o pooler em transaction-mode (a transação roda inteira num backend). A constraint é só defesa-em-profundidade. Mudança invasiva numa lógica de cupom correta. | Tabela contadora `coupon_user_usage` com `UPDATE … WHERE used < limit RETURNING` (igual ao `max_redemptions` global), ou índice único parcial p/ `per_user_limit=1`. |
| Ausência de rate limiting (enumeração de cupom, spam de checkout) | 🟠   | Precisa de **store compartilhado** (Upstash/Vercel KV/Redis); limiter em memória é inócuo em serverless multi-instância.                                                                                                                             | Token bucket por IP+rota via KV nas server actions sensíveis; padronizar a mensagem de cupom inválido vs. esgotado (decisão de UX).                                  |
| ISR no catálogo (`colecoes`)                                      | 🟡   | `revalidate=60` força geração estática em **build-time**, que consulta o DB; o build mock-first (CI usa `DATABASE_URL` dummy inalcançável) **quebraria**.                                                                                            | ISR com acesso a DB no build, ou cache de app (unstable_cache) com fallback, mantendo o build mock-first.                                                            |
| Guard de build: falhar se `NODE_ENV=production` sem Clerk         | 🟡   | **Quebraria o CI**: o build de CI usa chaves Clerk placeholder → `isClerkConfigured()` é `false`. Um guard duro reprovaria o build hermético.                                                                                                        | Checklist de deploy + verificação em runtime no boot de produção (não no build).                                                                                     |
| UI de checkout mostra total **sem** o cupom                       | 🟡   | Regra de frete no servidor já é correta e determinística (limiar sobre mercadoria, pré-cupom). Corrigir a UI exige uma server action de **preview de cupom** (a validação é server-only) — é trabalho de feature.                                    | Endpoint/preview server de cupom que devolve desconto + total final para a `CheckoutView` refletir.                                                                  |
| Reconcile re-busca `getOrderById` por pedido pago                 | 🟡   | 1 query extra por pedido pago reconciliado (lote ≤50), não um N+1 por item. Refatorar mexe na assinatura de retorno do caminho de pagamento.                                                                                                         | `applyPaymentStatusTx` retornar o `Order` (já seleciona `items`) p/ o e-mail, sem novo round-trip.                                                                   |
| `inventory.ts` faz 1 UPDATE por item                              | 🟡   | São N round-trips dentro de **uma** transação (N = itens do carrinho, tipicamente pequeno). Batch em `reserveStock` complica a identificação do item sem estoque, no caminho crítico recém-endurecido.                                               | `UPDATE … WHERE id = ANY($1::uuid[])` para release/commit/restock; `reserveStock` com `RETURNING` p/ achar o item que falhou.                                        |
| Rebaixamento de admin impossível via `ADMIN_EMAILS`               | 🟡   | Decisão de política. O padrão atual ("nunca rebaixa admin existente") é seguro/anti-flap.                                                                                                                                                            | Ação de admin auditada para alterar role; documentar que `ADMIN_EMAILS` concede, não revoga.                                                                         |
| `user.deleted` deixa `orders`/`redemptions` órfãos                | ⚪   | **Intencional/desejável**: preservar pedidos por questão fiscal/auditoria; FK-por-texto desacopla do Clerk.                                                                                                                                          | Sem mudança; documentado.                                                                                                                                            |
| Webhook sem teto explícito de payload                             | ⚪   | Limites de plataforma já cobrem.                                                                                                                                                                                                                     | `Content-Length` → 413 acima de um teto, se desejado.                                                                                                                |

## Como rodar as provas

```bash
# Postgres descartável + schema (ver tests/nplusone/README.md)
export TEST_DATABASE_URL="postgresql://postgres:test@HOST:5432/db"
pnpm test               # todas as suites (sem TEST_DATABASE_URL, as de DB são puladas)
```

`scripts/qa-gate.sh` continua verde sem banco (suites de DB são opt-in; `build`
exige apenas as envs dummy do CI).
